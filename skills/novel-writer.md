# 正文写作 (Novel Writer)

小说正文的完整写作流程。调用此技能前应已加载 worldbuilding、characters、outliner 等配套技能。

## 0. 黄金规则

- 写完一章后**必须先自查再提交**（见第8节），确认无矛盾后再交给用户审阅。自查是强制性的，不是可选项。
- 用户确认无误后，做一次**去AI味处理**（见第8节），不改变任何原文内容
- 用户确认最终版后，才能继续下一章
- **绝不跳步骤**：写 → 自查 → 用户初审 → 去AI味 → 用户终审 → 下一章
- **发现矛盾先自己修**，修完再提交。不要把矛盾留给用户来抓。
- **信息密度控制**：百万字级小说，第一章不宜展示过多设定。只放「这是什么世界、主角是谁」，深层伏笔留给后面几十章慢慢铺。一次释放的信息不要超过读者在当前阶段需要的量。
- **主动加载写作工具**：续写/创作任务启动时，先执行**前置步骤**（加载 novel-writer、novel-style-guide、novel-characters、novel-pacing 等配套技能并阅读对应章节），不得跳过。本次会话中因此出错——写了35章后才被用户指出叙事节奏问题，而 novel-style-guide 第7节早已记录了此约束。

## 1. 写作流程

### ⚠️ 前置步骤（强制 — 不执行即可能导致章节被退回）

在调用 novel-writer 进行任何写作之前，**必须先加载以下技能并阅读对应章节：**

| 技能 | 必须阅读的章节 | 原因 |
|------|---------------|------|
| `novel-style-guide` | 第7节「叙事平衡法则」(7.1/7.2/7.3) | 用户硬性文风约束，违反即退回 |
| `novel-characters` | 全部 | 确保角色行为不OOC |
| `novel-outliner` | 当前卷大纲 | 知道本章在全卷中的位置 |
| `novel-worldbuilding` | 当前地图/势力设定 | 避免设定冲突 |

**加载命令**（续写/创作任务启动时主动执行，不等用户提醒）：
```
skill_view(name='novel-writer')
skill_view(name='novel-style-guide')
skill_view(name='novel-characters')
skill_view(name='novel-pacing')
```

### 每章写作前的准备
1. 加载当前卷的大纲，确认本章在全卷中的位置和字数目标
2. 查看角色当前状态（境界、位置、情绪）
3. 确认本章需要达成的目标（推进剧情/战斗/情感/世界观展示）
4. 回顾上一章结尾，自然衔接
5. 确定本章的信息释放量——百万字级小说，第一章不宜展示过多设定

### 章节模板（纯文本格式）

```
[开头 100-200字]
用一个具体的场景或动作切入，不从背景介绍开始。

[主体 3000字左右]
对话+描写+推进交替，不用大段独白。
战斗、情感、反转分别调用对应技能。

[结尾 100-200字]
轻钩子，自然收束。
```

### 字数控制
- **硬性下限：≥2000字符**（统计方式：`wc -m <file>`，含标点、空格、换行——即文件内所有字符数。这是不可突破的约束，一次不足即触发修正）
- **无硬性上限**，以叙事完整性为准
- **优先级**：叙事完整性 > 字数达标。如果剧情在2000字处已有自然的收束点，不要强行扩写
- 章节末尾标注（第X章 完）
- **历史章节规则**：已被用户确认过的章节，不得再回头修改。所有修正只影响当前正在写的新章节。用户明确说「历史章节不要修改了」时，立即停止对任何旧文件的改动。

### 章节标题格式

每章在正文前加一行标题：

```
第X章 章节名

正文从新行开始……
```

- 标题与正文之间空一行
- 标题格式统一：「第X章 + 空格 + 章节名」
- 标题名应与本章核心内容相关，不宜过长（≤10字最佳）
- 章节名应在写完正文后根据实际内容拟定

## 2. 番茄小说格式规范

### 排版
- **纯文本，不加 markdown 标题符号**
- 段落短小，每段 3-4 行以内
- 段落之间空一行
- 首行不缩进，靠段落空行区分

### 对话
- 对话统一使用双引号 `"..."` 包裹
- 不用 `「」` 或其他引号符号
- 说话标签用冒号：`江寻问道："你吃了吗？"`
- 动作穿插对话时，动作描述在引号外

### 标点
- 省略号用 `……`（占2字符）
- 破折号用 `——`（占2字符）
- 避免连续使用 `！！！` 或 `？？？`
- 全篇统一用中文标点

## 3. 百万字长篇的首章节奏

> **参考文件**: `references/character-consistency.md` — 每章写完后的角色行为基线检查
 
对于预计超 100 万字的网文，第一章的作用只是「让读者知道这是什么世界、主角是谁」。

### 第一章可以放的
- ✅ 主角的日常状态和性格剪影
- ✅ 世界的一个横截面（通过日常细节自然带出）
- ✅ 1-2 个极轻的伏笔（只需让读者有印象）
- ✅ 一个简单的结尾钩子

### 第一章不要放的
- ❌ 完整的力量体系等级说明
- ❌ 复杂的势力格局
- ❌ 几十万字后才回收的深远伏笔
- ❌ 多个配角同时密集出场
- ❌ 大量世界观设定堆砌
- ❌ 过于明显的角色身份暗示

### 信息释放节奏建议
- 第一章：展示世界什么样、主角是谁
- 第2-3章：引入核心冲突
- 第3-5章：通过事件自然带出力量体系轮廓
- 前10章：让读者对整个世界有大致的了解

## 4. 角色一致性要求

### 核心原则
角色的每一个动作和对话，都必须符合该角色的人设基线。
**不能用「为了推动剧情」来合理化角色的OOC行为。**

### 常见陷阱
- ❌ **内向的角色突然主动找主角聊天**——除非有充分的动机铺垫
- ❌ **话少的角色长篇大论解释设定**——不说设定，用行动展示
- ❌ **冷静的角色第一次见到怪物就尖叫**——人设崩塌
- ❌ **为制造伏笔让角色说出不符合身份的话**——读者会出戏

### 检查方法
如果一段对话或行为让人感到「ta不像会说这种话/做这种事的人」，就说明OOC了。
宁可删掉这个信息点，也不要牺牲角色一致性。

### 已觉醒/未觉醒的区分
- 已经觉醒灵能的角色不需要参加天赋觉醒检测
- 已有检测结果的角色不能用「不知道结果如何」的表述
- 已觉醒者的检测相关对话必须与设定一致
- 两年未觉醒的主角，心态应是「习惯了」而不是「期盼」

## 5. 觉醒场景写作规范

初次觉醒（金手指激活）是全书第一个关键场景，需特别注意：

### Do's
- ✅ 觉醒过程**不可控**——主角无法理解正在发生什么
- ✅ 觉醒效果**有限**——初次觉醒是雏形，不是完全体
- ✅ 觉醒伴随着**身体不适应征**——心慌、出汗、呼吸急促、发抖
- ✅ 觉醒一闪即逝——主角努力尝试也无法立刻重现
- ✅ 让读者感知到「有东西觉醒了」，但不知道它具体能做什么

### Don'ts
- ❌ 初次觉醒就全场时停/大范围影响环境
- ❌ 主角立刻理解并控制新能力
- ❌ 觉醒后毫发无损、无缝衔接战斗
- ❌ 初次觉醒的威力过于逆天，削弱后续成长空间

### 觉醒后的状态模板
```
觉醒前的危机 → 体内异样感（震动/发热/发凉）
→ 感知异常（时间变慢/视觉清晰/声音过滤）
→ 主角处于被动接受状态，无法操控
→ 状态自动消退 → 残留体征（手抖/出汗/心跳加速）
→ 主角知道发生了什么，但不理解
```

## 6. 每章必查清单

- [ ] 开头从具体场景切入（不从背景介绍开始）
- [ ] 前 100 字是否有钩子
- [ ] 字数在目标范围内（过渡章≥2000，标准章≥3500）
- [ ] 所有角色说话和行动符合人设
- [ ] **叙事平衡法则合规**：无连续5句以上纯叙事；每段≤5句；有内心活动或对话打散叙事密度

- [ ] 纯文本格式，无 markdown 残留

## 7.5 续写中的重大设定变更处理

续写过程中，用户可能要求对某些设定进行**全局性修正**（如某势力已覆灭不再提及、某角色能力问题已解决不再重复）。这种修正不同于章节内的错别字，影响面可能覆盖多个已写章节。

### 发现信号

- 用户明确说「XX已经不复存在，后续不要再有提及」
- 用户说「XX的问题已经处理，不要再提及」
- 用户的纠正是针对**世界观规则**而非语言表达

### 处理流程

1. **立即停止当前写作**，先做全局扫描确认受影响范围
2. **删除或重写受影响的章节文件**（不能保留错误设定待后续修正——已写的就是错的）
3. **重新合并数据并重启服务器**，确保前端展示的是修正后的版本
4. **在记忆中将该设定标记为已处理**，避免后续写作中再次触碰
5. **恢复写作后，在自查环节额外检查一次**：新写的内容中是否无意间触碰了已删除的设定

### 常见错误

- ❌ 试图保留旧章节只修改后续——犯错的章节本身就是错的，必须重写
- ❌ 改了一章却忘了查同一卷中其他章节是否也涉及该设定
- ❌ 用户说了「不要提及」之后，仍用同义词或相关概念暗示该设定

## 8. 章节自查流程（四维自查 — 强制步骤 — 写完自动执行）

写完一章后，**在交给用户审阅之前**必须自动执行四维自查。**不要等用户来问「自查了吗」，而是主动在写完内容后：①执行自查 → ②修正问题 → ③输出结构化自查报告 → ④将报告与正文一并提交给用户。**

用户对此有明确的预期：每章写完自动自查，不需用户提醒。

### 四维自查框架

四维自查是用户明确要求的检查标准，覆盖四个维度：**设定冲突 → 前后矛盾 → 错别字 → 啰嗦篇幅**。按此顺序执行，每个维度独立检查。

```
=== 四维自查 ===

1️⃣【设定冲突】检查
   - 世界观设定与本卷/全局设定是否一致
   - 力量体系是否与之前展示的一致（境界、能力效果、突破条件）
   - 新引入的设定是否有原作线索支撑（续写场景下）
   - 角色实力是否与之前展示的对得上
   - 角色行为是否符合该角色的人设基线
   - 已觉醒角色是否被错误地描绘为未觉醒？
   - 角色说话语气是否前后一致？

2️⃣【前后矛盾】检查
   - 本章开头与上一章结尾能否自然衔接
   - 时间线是否无误（季节、日夜、事件先后顺序）
   - 位置逻辑正确（角色从上章结尾位置出发）
   - 角色状态与上章结尾一致（伤势、修为、装备）
   - 续写内容与原文设定不冲突
   - 新引入的伏笔不覆盖/修改已有伏笔的走向

3️⃣【错别字】检查
   - 常见形近字：的/地/得、在/再、他/她/它
   - 专有名词是否写错（人名、地名、功法名）
   - 引号全部为双引号 " "（不是「」或『』）
   - 无 markdown 符号残留（# ## *** 等）
   - 标点符号：省略号「……」（占2字符）、破折号「——」（占2字符）
   - 段落长度合理（每段3-4行以内）

4️⃣【啰嗦篇幅】检查
   - 同一信息是否被重复表达（如「A感觉奇怪……但说不上来哪里奇怪」→ 合并）
   - 是否有多余的心理独白（能用动作/对话体现的，删掉独白）
   - 是否有多余的过渡语（「与此同时」「就在这时」→ 直接写下一件事）
   - 是否存在连续同句式（如「好像……好像……」→ 改一个）
   - 是否存在大段背景说明（超过5行就需要拆成事件去展示）
   - 字数检查：过渡章≥2000，标准章≥3500（wc -m统计总字符）
```

### 四维自查执行顺序（必做 — 不可跳步）

完整流程：**写正文 → 执行四维自查 → 修正发现的问题 → 输出自查报告 → 更新数据 → 交付给用户**

**第一步：立即检查字数。** 在进入四维自查之前，先用 `wc -m <file>` 统计字符数。若低于 2000，**先扩写至达标**再进入自查。不要带着字数不足的问题进入自查流程——字数不足本身就是不合格，不需要自查其他维度。

**第二步：执行四维自查。** 按 设定冲突 → 前后矛盾 → 错别字 → 啰嗦篇幅 的顺序逐一检查。

**第三步：修正问题。** 每个维度发现的问题，修正后方可进入下一维度。

**第四步：输出自查报告（强制 — 不可省略）。** 以结构化表格输出（格式见下方），与正文在同一轮交付给用户。**用户明确要求：写完正文后必须先输出结构化自查报告才能提交，不要等用户来问「自查了吗」。自查报告的三个约束：①必须输出——不能只做自查不写报告；②必须结构化——使用下方表格格式，不要大段文字描述；③必须与正文同一轮交付——不要在用户看完正文后才补报告。**

### 四维自查报告格式（强制输出）

自查完成后，必须以结构化表格的形式输出报告，放在正文之后与正文同一轮交付。格式如下：

```
**四维自查报告：第X章**

| 维度 | 检查项 | 结果 |
|------|--------|------|
| 1️⃣ 设定冲突 | [列出具体检查条目] | ✅/❌ |
| 2️⃣ 前后矛盾 | [列出具体检查条目] | ✅/❌ |
| 3️⃣ 错别字 | [列出具体检查条目] | ✅/❌ |
| 4️⃣ 啰嗦篇幅 | 字数≥2000（wc -m） | ✅ [实际字符数] |
| | [其他条目] | ✅/❌ |

**自查结论：通过/未通过**
```

**注意事项**：
- 每个维度至少覆盖2-3个检查条目，不能只写一行"无问题"就跳过
- 字数必须列明 `wc -m` 的实际数值，不能只写"达标"
- 若自查结论为"未通过"，报告中必须说明修正了哪些问题
- 字数不足时（<2000），先补充内容至达标，再输出报告

### 续写场景下的额外检查

如果是在续写已有小说的场景下，需额外增加：

- 续写中引入的新设定不覆盖/修改原文已有设定
- 原文角色的实力线和性格线不被续写暗中改动
- 新章节的语气/节奏与原文一致（除非剧情需要刻意变化）
- 原文未闭合的伏笔在续写中被合理接收

## 9. 去AI味处理规范

用户确认初审通过后，执行去AI味处理。核心原则：**不改变任何原文内容**（剧情、对话、角色关系、场景顺序不动），仅调整语言的自然度。

### 常见AI味特征（需替换的）
- 「的时候」→ 可省略为「时」
- 「他的/她的」→ 上下文清晰时省略
- 「经过/路过/走进……的时候」→ 去掉「的时候」
- 「他知道/他感觉/他似乎」→ 能通过动作体现时删去
- 「忽然/突然/猛然」→ 直接写动作即可
- 「似乎/好像/仿佛」→ 删减 ≈50%
- 「但/不过/然而」→ ≈50% 可去掉
- 过于整齐的排比结构 → 打散
- 过多「他/她」开头 → 调整主语
- 角色全名反复出现 → 后续用「他」「江寻」交替

### 易bug的句式（写时警惕，查时重点检查）

详细案例见 `references/selfcheck-bugs-found.md`（本会话自查记录）。

- **双重否定病句**：如「也不不紧张」→ 应改为「说不上紧张，也说不上不紧张」
- **冲突的外貌描写**：不能同时说一个人「扎着马尾」和「短发」
- **重复用词**：如「有点奇怪……哪里奇怪」→ 改为「有点怪……哪里不对」
- **缺逗号的长句**：如「过了不到一分钟出来了」→ 加逗号「过了不到一分钟，出来了」
- **同场景主语混淆**：同段中A在转笔，别写B也在转笔——分清主语

### 保留不动的内容
- ✅ 所有对话原文（一字不动）
- ✅ 所有剧情节点
- ✅ 所有角色之间的互动
- ✅ 所有场景描写
- ✅ 所有伏笔和暗示

### 字数监控
- 纯中文字数不应低于去味前的 85%
- 如果字数下降过多，说明过于激进，需恢复部分内容
- 目标：在保持自然的前提下尽量保留原文

## 10. 续写已有小说的工作流

续写他人已创作的小说（非自己原创），与原创写作的流程有所不同。关键区别在于：**先分析再动笔，匹配原有文风而非自由发挥。**

### 10.1 读取阶段（EPUB提取）

第三方小说通常以 EPUB 格式交付。无需专门库，用 Python 内置模块即可提取：

```python
import zipfile, os, tempfile, re

# EPUB = ZIP 文件
tmpdir = tempfile.mkdtemp()
with zipfile.ZipFile(epub_path, 'r') as z:
    z.extractall(tmpdir)

# 找到所有 XHTML 章节文件
xhtml_files = sorted([f for f in os.listdir(...) if f.endswith('.xhtml')])

# 去除 HTML 标签提取纯文本
text = re.sub(r'<[^>]+>', '', content)
text = re.sub(r'\n\s*\n', '\n\n', text)
```

详见 `references/epub-extraction.md`。

### 10.2 分析阶段（续写前必须完成的准备）

读取完毕后，整理以下信息结构：

1. **基本信息**：书名、作者、字数、章节数、卷结构
2. **主角档案**：姓名、背景、年龄、能力体系、核心目标
3. **重要角色图谱**：亲人层、伴侣线、师门层、同门层、反派层、特殊存在
4. **世界观架构**：力量体系层次、世界层级结构
5. **情节脉络**：按阶段划分，标注每个阶段的起止章节和核心事件
6. **未完线索清单**：所有未闭合的悬念、伏笔、角色命运
7. **写作风格分析**：视角、对话格式、描写密度、战斗系统、感情线处理方式

### 10.3 匹配文风（关键步骤）

- **视角**：确定第三人称有限视角还是上帝视角
- **对话格式**：双引号/单引号，说话标签格式（冒号/逗号）
- **描写密度**：环境描写占比、对话与叙述的比例
- **战斗系统**：招式命名风格、境界描写方式、战斗节奏
- **感情线处理**：克制含蓄还是直白表达、进展速度

### 10.4 续写衔接规范

- **开头回稳**：续写首章前200-300字通过环境/对话/动作让读者确认还在同一个故事里
- **老角色推进**：先让已有角色行动，再通过互动自然引入新元素
- **新元素引入**：每次不超过1-2个新信息点
- **伏笔接力**：确认已有伏笔状态，在此基础上编织新伏笔
- **文体一致性**：优先匹配原文风格，而非「优化」它

### 10.5 续写中的角色一致性

- **不能修正原角色**——原作者设定就是最终设定
- **不能调整实力设定**——原书已展示的境界和战力不能暗中改动
- **补全需有依据**——揭示原作未写明的设定，必须有原作线索支撑

### 10.6 用户提供的世界观设定修正

续写过程中，用户可能随时给出**新的世界观设定**或**修正已有设定**（如某势力已覆灭、某角色真实身份、某件物品的来历等）。这些信息不是可选的背景知识，而是必须在后续写作中准确遵循的硬性约束。

#### 接收流程

当用户给出设定信息时：

1. **立即保存到记忆**，使用 `memory(action='add')`
2. **检查已有续写章节**是否已经触犯了该设定（写了与设定矛盾的内容）
   - 如果发现矛盾，**立即删除或重写**受影响的章节，不能留着等以后修
   - 重写后更新数据文件并重启服务
3. **在后续写作中，在自查环节增加一条**：检查新写内容是否与该设定一致

#### 常见陷阱

- ❌ 用户给了设定，但忘记在写作中体现（如：破晓剑说雷域中心留了东西 → 后续写到雷域时没去找）
- ❌ 用户修正了设定后，仍沿用旧设定写作（如：用户说万雷谷已覆灭 → 但仍写"林一前往万雷谷"）
- ❌ 在一章中同时使用新旧设定导致矛盾

#### 记忆 vs 技能的区分

| 存储位置 | 适合存放的内容 |
|---------|--------------|
| 记忆 (memory) | 用户提供的具体设定、角色关系、世界细节、伏笔方向 |
| 技能 (skill) | 处理这类设定的通用方法、流程、检查清单 |

用户给的设定细节保存在记忆中；记住要去核查这些设定的方法保存在技能中。

### 10.7 场景/势力/角色关系的状态核查（续写特有 — 高频坑点）

**本节的坑点是本会话中真实踩过的。** 用户纠正过的地方，不是可以绕开的偶然错误，而是续写时必然再次出现的系统性问题。以下规则每写一章前至少扫一遍。

续写时，**不要假设你记忆中的任何地点、势力或角色关系仍然存在**。在广元大陆→七情界的过渡中，许多原广元大陆的宗门已经不复存在。这是一个高频坑点。

#### 每次提到某个地点或势力之前，先自问：

- 这个地方/势力在**当前时间线**上还存在吗？（如：万雷谷在阵法破碎后已覆灭）
- 它是否已被其他势力吞并、瓜分或取代？
- 它的覆灭是否在本作中有明确交代？（如果有，不能自行设定它还活着）
- 角色对该地点/势力的称呼是否准确？（如：不能说"去万雷谷"，只能说"去雷域"）

#### 每次提到某个角色关系之前，先确认：

- 该角色称对方为什么？（师父？师叔？师兄？）
- 这个称呼是否与之前的设定一致？
- 是否有用户最近修正过的关系？（如：林紫萱的师父是念若尘，应称"师父"而非"师叔"）

#### 预防措施

- 在写作前，快速扫描记忆中的相关条目
- **硬性规则**：不要假设你记住了所有设定。每个地点/势力在提到之前，先确认它的当前状态。写雷域之前先确认万雷谷是否还存在（已覆灭），写林紫萱的称呼之前先确认她的师父是谁（念若尘，不是师叔），写破晓剑鞘之前先确认剧情时间点是否合适（飞升前，不是现在）。
- 如果某个地点在一段时间内没有被提及过，先查证它的当前状态再用
- 自查时增加一条："所有提及的地点/势力在当前时间线上是否存在"

### 10.8 续写数据管理

续写时通常需要将原始章节（来自 EPUB）与续写章节（来自 TXT 文件）合并为一个 JSON 文件供 Web 应用使用。这是数据操作，但操作失误会导致原始数据丢失。

**关键原则**：永远不要完全替换 JSON 文件的 `chapters` 数组。只追加新章节，保留旧章节。

**正确做法**：读取已有 JSON → 扫描续写目录 → 只追加不在 JSON 中的新章节 → 按 id 排序 → 写回。

**常见错误**：清空 `chapters` 数组后只从 TXT 文件重新构建，导致原始 196 章丢失。恢复方法见 `references/merge-workflow.md`。

**验证命令**：
```bash
python3 -c "import json;d=json.load(open('novel_data.json'));print(f'总章节:{d[\"total_chapters\"]}, ids:{min(c[\"id\"] for c in d[\"chapters\"])}-{max(c[\"id\"] for c in d[\"chapters\"])}')"
```

## 11. 主线进度报告（写完一章后输出）

写完一章并完成自查后，用户期望你主动输出 **主线进度报告**，告知当前故事整体进展。

### 报告内容

1. **当前卷/幕**的名称和完成百分比
2. **各幕状态**（已完结/进行中/未开始），进行中的幕标注进度百分比
3. **本章位置**：属于哪一幕的哪个阶段
4. **下一阶段预告**：接下来1-3章将推进什么
5. **全卷预估**：当前大约完成了全卷的百分之多少

### 报告格式

```
**主线进度**

[卷名] 约完成 [X]%

- **[幕名]**（第X-X章）→ 已完结
- **[幕名]**（第X-X章）→ 已完结
- **[幕名]**（第X-X章）→ 进行中，约 [X]%
  - 当前节点：[一句话描述当前剧情位置]
  - [下一阶段预告]

全卷预估：约 [X]%
```

### 输出时机

放在自查报告之后，与正文和自查报告同一轮交付给用户。不需要等用户问。

## 13. 自动创作定时任务（Cron Auto-Write）

当用户要求将写作模式从手动切换为自动批量创作时，配置 cron 定时任务。该 cron 每天凌晨自动生成 6 个章节，并自动执行自查和上传。

### 13.1 已配置的 cron 任务

| 项目 | 内容 |
|------|------|
| 任务名 | `novel-auto-write` |
| 定时 | 每天 `00:00` (cron 表达式 `0 0 * * *`) |
| 产量 | 每次 6 章 |
| 工作目录 | `/root/hermes-agent-novel/` |
| 上下文脚本 | `scripts/novel-context.sh`（技能目录下，部署时复制到 `~/.hermes/scripts/`） |
| 加载的技能 | novel-writer, novel-style-guide, novel-characters, novel-pacing |
| 上传 API | 见 `references/upload-api.md` |

### 13.2 Cron 工作原理

每次运行时：

1. **运行上下文脚本** `scripts/novel-context.sh`，自动收集：
   - 最新章节号（从 `chapter-XXX.md` 文件命名解析）
   - 大纲（`outline.md`，头80行 + 剩余行）
   - 世界观设定（`worldbuilding.md`，头100行）
   - 角色档案（`characters.md`，头100行）
   - 最近2章完整原文（用于风格衔接）
   - 情节追踪汇总（`plot-tracking.md` 的汇总表和进行中的暗线/伏笔）
   - 最新一章字数统计

2. **加载写作技能**：`novel-writer`, `novel-style-guide`, `novel-characters`, `novel-pacing`

3. **逐章生成**，每章完成以下步骤：
   - 读取上一章结尾确认衔接点
   - 阅读 plot-tracking.md 中当前进行中的暗线和伏笔，在写作中自然推进或呼应
   - 写出正文（段落控制 ≤ 5句/段，心理活动打散叙事密度）
   - 保存到 `/root/hermes-agent-novel/chapter-NNN.md`
   - 四维自查（设定冲突、前后矛盾、错别字、啰嗦篇幅）
   - 字数不足 2000 时补充内容
   - 去AI味处理（不改变剧情内容）
   - **更新 plot-tracking.md**：如果有新的伏笔、暗线推进、事件发生、状态变化，更新对应条目
   - 上传到 API

4. **循环步骤 3** 直到完成 6 章

5. **输出总结**：完成的章节范围、字数、故事进度、本次新增/更新的伏笔和暗线状态变化

### 13.3 cron prompt 编写要点

cron prompt 必须包含以下硬性约束：
- 段落控制（每段 ≤ 5句）
- 心理活动穿插（不能连续 5 句纯叙事）
- 番茄格式（纯文本、双引号对话、无 markdown）
- 字数 ≥ 2000（`wc -m` 统计）
- 每章自查（四维自查框架 + 结构化自查报告输出）
- 去AI味处理
- 更新 plot-tracking.md
- 上传到 API

### 13.4 Cron 管理命令

```bash
# 查看 cron 任务列表
hermes cron list

# 查看具体任务
hermes cron show novel-auto-write

# 暂停（不删除）
hermes cron pause novel-auto-write

# 恢复
hermes cron resume novel-auto-write

# 删除
hermes cron delete novel-auto-write

# 手动触发一次（用于测试）
hermes cron trigger novel-auto-write
```

上下文收集脚本 `scripts/novel-context.sh` 输出以下内容（被 cron prompt 直接读取）：

## 14. 情节追踪管理（Plot Tracking）

当长篇小说体量超过 30 章时，手动记忆所有伏笔、暗线和事件状态变得不可靠。需要维护一份 `plot-tracking.md` 作为创作期间的情节「仪表盘」。

### 14.1 追踪内容

| 类别 | 描述 | 状态字段 |
|------|------|---------|
| 主线 (Main Plot) | 故事的核心推进线 | 已揭示 / 进行中 / 已回收 |
| 暗线 (Hidden Plot) | 读者尚未完全知晓的秘密线 | 未触发 / 进行中 / 已揭示 |
| 伏笔 (Foreshadowing) | 埋下的种子，等待后续回收 | 未触发 / 等待揭示 / 进行中 / 已回收 |
| 支线 (Side Plot) | 丰富世界观的次要故事线 | 未触发 / 进行中 / 已回收 |
| 重大事件 (Major Events) | 已发生的关键节点 | ★~★★★★★ 按影响程度分级 |

### 14.2 文件结构

参考 `references/plot-tracking-template.md`。模板包含：

1. **主线表** — 每条记录：序号、主线名、起始章、当前状态、进度说明
2. **暗线表** — 同上格式
3. **伏笔表** — 每条记录：序号、伏笔名称、埋下章、当前状态、说明
4. **支线表** — 同上格式
5. **重大事件年表** — 按时间顺序排列：序号、事件、章节、重要性(★)、影响
6. **汇总表** — 各类别按状态维度的数量分布

### 14.3 维护规则

**何时更新**：
- 每写一章后，检查本章是否推进了任何一条线
- 引入新的伏笔或暗线时，立即在对应表中新增一行
- 状态变更时（如未触发→进行中，进行中→已揭示），更新状态字段和进度说明
- 重大节点发生时，在重大事件年表中追加一行

**状态转换规则**：

```
未触发 → 进行中 → 已揭示/已回收
                  ↕
              等待揭示（伏笔特有）
```

### 14.4 与自查的联动

在四维自查的【设定冲突】检查中，增加一条：
- 新写的内容是否与 plot-tracking.md 中记录的已回收/已揭示状态一致？
- 不要重复揭示已经揭示过的内容

### 14.5 与 Cron 自动创作的联动

cron 任务的每一步都必须包含：
1. 读取 plot-tracking.md 了解当前进行中的线
2. 在创作中自然推进或呼应当前活跃的线
3. 写完后更新 plot-tracking.md：新增/推进/回收，同时更新汇总表底部的统计数据

**不允许**：cron 写入的章节与 plot-tracking.md 的记录状态不同步。

## 15. 创作系统仓库化（Novel Writing System Repo）

当写过多部小说后，技能、模板、脚本和流程应当组织为一个独立仓库，方便复用。见 `references/novel-repo-organization.md`。
